Проект ОТЧЕБОТ (@otche_bot) – Полная архитектурная спецификация

1. Research
1.1 Назначение проекта
Telegram-бот для сбора «исповедей» – свободных текстовых описаний проблем пользователей в IT-сфере. 
Бот выступает промежуточным звеном между пользователем и администратором (или внешней системой),
аккумулирует заявки и предоставляет к ним доступ через API для дальнейшей обработки (категоризация, помощь специалистов).

1.2 Целевая аудитория и нагрузка
Канал с 10 000 подписчиков.

Пиковая нагрузка: 300 пользователей в час (до 5–10 одновременных запросов, но возможны кратковременные всплески до 20–30).

Ожидаемое количество заявок: 500–1500 в день (конверсия 5–15%).

Объём данных в год: ~1 ГБ (с индексами).

1.3 Функциональные требования
1.3.1 Взаимодействие с пользователем
Вход: Пользователь переходит по ссылке из канала и нажимает /start.

Приветствие: Бот отправляет сообщение в светлом дизайне («Золотая прозрачность») с кнопкой «Начать исповедь».

Форма ввода: После нажатия кнопки бот переводит пользователя в состояние ожидания текста (FSM). Пользователь отправляет сообщение (до 500 символов).

Валидация: Проверка длины и непустоты. При ошибке – повторный запрос.

Согласие: После корректного текста бот запрашивает согласие на сбор данных профиля (ID, username, имя, фото) с помощью инлайн-кнопок «Согласен» / «Отмена».

Сбор данных: При согласии бот через Telegram API получает:

user_id

username (если есть)

first_name, last_name

photo_file_id (самое большое фото профиля, если есть)

Сохранение: Данные и текст сохраняются в БД (PostgreSQL) с отметкой времени и статусом new.

Ответ пользователю: Бот отправляет сообщение: «Ваша заявка будет обработана в ближайшее время. Сейчас бот в тестовом режиме, время обработки до 2 суток. Спасибо за понимание.»

Право на забвение: Команда /delete_my_data удаляет все заявки пользователя из БД.

1.3.2 Админ-панель
Доступ: Только пользователи, чьи telegram_id перечислены в переменной окружения ADMIN_IDS (список через запятую). Смена администратора – вручную через редактирование .env и перезапуск.

Команда /admin: Отображает меню с инлайн-кнопками:

«📨 Новые заявки» – просмотр заявок со статусом new (пагинация).

«📤 Отметить как обработанное» (для каждой заявки).

«📊 Статистика» (количество заявок за период).

(Опционально) «📦 Выгрузить в API» – триггер для pull-механизма (см. ниже).

Просмотр заявки: Для каждой заявки показывается карточка с данными пользователя (имя, юзернейм, фото (если есть), текст заявки) и кнопками для изменения статуса.

1.3.3 Передача данных внешнему агенту (pull-механизм)
Механизм: Внешний агент делает HTTP GET-запрос к эндпоинту /api/v1/problems с заголовком авторизации X-API-Key: <ключ>.

Параметры запроса: ?status=new&limit=100&offset=0 (пагинация, статус по умолчанию new).

Ответ: JSON-массив объектов, каждый содержит:

{
  "id": 1,
  "user_id": 12345,
  "username": "durov",
  "first_name": "Pavel",
  "last_name": "Durov",
  "photo_file_id": "AgACAgIAAxkB...",
  "text": "У меня не работает принтер",
  "status": "new",
  "created_at": "2024-01-01T12:00:00Z"
}


Пометка отправленных: Эндпоинт POST /api/v1/problems/{id}/mark_sent позволяет пометить конкретную заявку как sent. Либо можно передать параметр mark_as_sent=true в GET-запросе, чтобы автоматически сменить статус у выгруженных записей (реализуем оба варианта на выбор администратора).

Безопасность: API-ключ хранится в переменной окружения EXTERNAL_API_KEY. Эндпоинт доступен только по HTTPS (настройка сервера).

1.3.4 Антифлуд
С одного аккаунта Telegram не чаще 1 заявки в минуту.

Реализация: in-memory словарь {user_id: last_timestamp}, сбрасываемый при перезапуске. При превышении лимита – сообщение пользователю и игнорирование.

1.3.5 Репликация и бэкапы
Репликация: Настроить асинхронную стриминг-репликацию PostgreSQL на резервный сервер (горячий standby).

Бэкапы: Ежедневные полные бэкапы через pg_dump (или pg_basebackup для физического) на резервный сервер. Хранить 7 дней.

Скрипты: Разработать интерактивные скрипты для настройки репликации и бэкапов, которые запрашивают у администратора:

IP резервного сервера

Путь к SSH-ключу

Путь для хранения бэкапов

(Опционально) параметры подключения

1.4 Нефункциональные требования
Технологический стек: Python 3.10+, aiogram 3.x, FastAPI, PostgreSQL 14+, SQLAlchemy 1.4+ (async), asyncpg, pytest.

Асинхронность: Весь код должен быть асинхронным (aiogram + FastAPI + asyncpg).

Безопасность: Все секреты в .env, API-ключ в заголовке, доступ по HTTPS.

Мониторинг: Логирование без персональных данных, возможно добавление метрик (Prometheus) на следующих этапах.

Масштабируемость: Архитектура должна позволять горизонтальное масштабирование в будущем (добавление реплик БД, кэширование Redis) без переписывания ядра.

2. Design
2.1 C4-модель (текстовое описание)
Контекст
Пользователь взаимодействует с Telegram Bot.

Администратор взаимодействует с Telegram Bot.

Внешний агент взаимодействует с API Server (pull).

Telegram Bot и API Server используют PostgreSQL Database.

Backup Server получает реплику и бэкапы от основного сервера БД.

Контейнеры
Telegram Bot Application – асинхронное Python-приложение на aiogram.

API Server – FastAPI приложение для внешних агентов.

PostgreSQL Database – основная реляционная БД.

Backup Server – отдельный сервер с PostgreSQL в режиме standby и местом для бэкапов.

Компоненты (внутри каждого контейнера)
Telegram Bot:

Handlers – обработчики команд и сообщений.

Form Service – управление FSM, валидация, антифлуд.

User Data Collector – получение данных профиля через Telegram API.

Admin Panel – логика админки.

Database Layer – асинхронные CRUD (SQLAlchemy + asyncpg).

API Server:

Routes – эндпоинты для внешних агентов.

Auth – проверка API-ключа.

Database Layer (аналогичный, или общий код).

2.2 Архитектурный подход: модульный монолит + Docker
После анализа требований и нагрузки принято решение использовать модульную архитектуру внутри монолитного репозитория, 
но с разделением на два отдельных процесса (бот и API), взаимодействующих через общую базу данных. Это обеспечивает:

Простоту и скорость разработки.

Лёгкость развёртывания и отладки.

Возможность в будущем выделить компоненты в независимые сервисы без переписывания кода.

Для унификации окружения и упрощения развёртывания рекомендуется использовать Docker-контейнеры:

Контейнер с ботом (на основе aiogram)

Контейнер с API-сервером (FastAPI)

Контейнер с PostgreSQL (или БД на хосте)

(Опционально) контейнер с Nginx для HTTPS и reverse proxy

Оркестрация – docker-compose для локальной разработки и продакшена. Это даёт изоляцию, воспроизводимость и лёгкость масштабирования
(можно запустить несколько экземпляров бота, используя общую БД и Redis для антифлуда при необходимости).

2.3 ADR (Architecture Decision Records)
ID	Решение	Обоснование
ADR-001	aiogram 3.x	Асинхронность, встроенная FSM, активная поддержка.
ADR-002	PostgreSQL + asyncpg	Конкурентность, репликация, надёжность.
ADR-003	pull-API (внешний агент запрашивает)	Контроль доступа, не требует инициативы от бота.
ADR-004	Хранение только file_id фото	Экономия места, соответствие политикам Telegram.
ADR-005	Секреты в .env, API-ключ в заголовке	Безопасно, стандартно.
ADR-006	Полностью асинхронный код	Высокая производительность, неблокирующий ввод-вывод.
ADR-007	Пул соединений к БД (min 10, max 30)	Эффективное использование ресурсов.
ADR-008	Антифлуд in-memory (словарь)	Достаточно для текущей нагрузки, простота.
ADR-009	Репликация (streaming) + ежедневные бэкапы	Требование отказоустойчивости и восстановления.
ADR-010	Статический список админов в .env	Простота, безопасность (изменение только через сервер).
ADR-011	Модульный монолит с Docker-контейнеризацией	Баланс простоты разработки и гибкости развёртывания; отсутствие необходимости в микросервисах при текущей нагрузке.

2.4 DFD (упрощённая диаграмма потоков данных)

Пользователь ──(текст, команды)──> Telegram Bot (контейнер)
Telegram Bot ──(запрос фото)────> Telegram API (внешний)
Telegram Bot ──(запись/чтение)──> PostgreSQL (контейнер или хост)
Администратор ──(команды)───────> Telegram Bot
Внешний агент ──(GET /api/v1/problems)──> API Server (контейнер)
API Server ──(чтение)────────────> PostgreSQL
Backup Script (cron) ──(pg_dump/replication)──> Backup Server

2.5 Sequence Diagrams
Сценарий 1: Исповедь пользователя

User -> Bot: /start
Bot -> User: Приветствие + кнопка «Начать»
User -> Bot: Нажать «Начать»
Bot -> User: Введите вашу проблему (до 500 символов)
User -> Bot: Текст проблемы
Bot -> Bot: Проверка длины, антифлуд
Bot -> User: Запрос согласия (кнопки)
User -> Bot: «Согласен»
Bot -> Telegram API: getChat, getUserProfilePhotos
Bot -> PostgreSQL: INSERT complaint (status=new)
Bot -> User: Сообщение о тестовом режиме

Сценарий 2: Админ просматривает заявки

Admin -> Bot: /admin
Bot -> Bot: Проверка ID в ADMIN_IDS
Bot -> Admin: Меню админки
Admin -> Bot: «Новые заявки»
Bot -> PostgreSQL: SELECT * FROM complaints WHERE status='new' ...
Bot -> Admin: Карточка заявки + кнопка «Отметить как обработанное»
Admin -> Bot: «Отметить»
Bot -> PostgreSQL: UPDATE complaints SET status='processed' WHERE id=...
Bot -> Admin: Подтверждение

Сценарий 3: Внешний агент получает данные

Agent -> API Server: GET /api/v1/problems?status=new&limit=10&offset=0
API Server -> API Server: Проверка API-ключа
API Server -> PostgreSQL: SELECT ... WHERE status='new' LIMIT 10 OFFSET 0
API Server -> Agent: JSON-массив
Agent -> API Server: POST /api/v1/problems/123/mark_sent
API Server -> PostgreSQL: UPDATE complaints SET status='sent' WHERE id=123
API Server -> Agent: 200 OK

2.6 Спецификация API (OpenAPI)

openapi: 3.0.0
info:
  title: OTCHÉBOT API
  version: 1.0.0
servers:
  - url: https://your-domain.com
security:
  - ApiKeyAuth: []
components:
  securitySchemes:
    ApiKeyAuth:
      type: apiKey
      in: header
      name: X-API-Key
paths:
  /api/v1/problems:
    get:
      summary: Get problems
      parameters:
        - name: status
          in: query
          schema:
            type: string
            enum: [new, sent]
          description: Filter by status (default new)
        - name: limit
          in: query
          schema:
            type: integer
            default: 100
            maximum: 500
        - name: offset
          in: query
          schema:
            type: integer
            default: 0
        - name: mark_as_sent
          in: query
          schema:
            type: boolean
          description: If true, automatically mark returned problems as 'sent'
      responses:
        '200':
          description: List of problems
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/Problem'
        '401':
          description: Invalid API key
  /api/v1/problems/{id}/mark_sent:
    post:
      summary: Mark a specific problem as sent
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: Status updated
          content:
            application/json:
              schema:
                type: object
                properties:
                  status: {type: string, example: 'sent'}
        '404':
          description: Problem not found
components:
  schemas:
    Problem:
      type: object
      properties:
        id: {type: integer}
        user_id: {type: integer}
        username: {type: string}
        first_name: {type: string}
        last_name: {type: string}
        photo_file_id: {type: string}
        text: {type: string}
        status: {type: string, enum: [new, sent, processed]}
        created_at: {type: string, format: date-time}

4.2 Рекомендации по безопасности
Никаких секретов в коде. Все переменные окружения загружаются из .env.

.env не коммитить, добавить в .gitignore.

Валидация всех входных данных от пользователя.

Параметризованные запросы (SQLAlchemy ORM).

Логировать только ID, не хранить персональные данные в логах.

API-эндпоинт доступен только по HTTPS (настроить Nginx reverse proxy с SSL в docker-compose).

- [ ] **Тесты**: Покрытие 70%+ с pytest
- [ ] **Webhook mode**: Альтернатива polling для продакшена

### Известные ограничения

1. **CHANNEL_ID** хранится в `.env` — при смене канала нужно обновить вручную
2. **Антифлуд in-memory** — сбрасывается при перезапуске бота (для масштабирования перенести в Redis)
3. **Нет rate limiting** на API — внешний агент должен соблюдать паузы
